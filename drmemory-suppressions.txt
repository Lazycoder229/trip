# 
# Dr. Memory suppressions - trip.exe
# 
# These suppress KNOWN FALSE POSITIVES that come from libraries trip.exe
# links against (OpenSSL's hand-optimized asm, mingw CRT internals), NOT
# from trip's own code (memory.c/object.c/table.c/tvm/*/compiler/*).
#
# Origin: full `make drmemory` run against tests/test_modules, 2026-07-22.
#   - 15 unique "unaddressable access beyond top of stack" errors, all
#     inside libcrypto-3-x64.dll / libssl-3-x64.dll (OpenSSL) or
#     __mingw_pformat (mingw's printf internals) - hand-optimized asm
#     stack probing that Dr. Memory's stack-tracking doesn't model
#     correctly. Well-known pattern when linking OpenSSL.
#   - 16 unique "uninitialized read" errors, all inside stat()/readdir()
#     by way of Windows FindFirstFileA/FindNextFileA - Dr. Memory doesn't
#     fully know which fields of WIN32_FIND_DATA the kernel actually
#     initializes, so it flags some as uninitialized even though they're
#     valid.
#
# Note: no trailing "..." after the last listed frame in each entry - Dr.
# Memory suppression matching is already a PREFIX match (you only need to
# list enough top frames to uniquely identify the pattern; any frames
# below what you list are automatically allowed to be anything). A
# trailing "..." after the last frame is what caused the "Malformed
# suppression" abort - it's only meant for skipping frames in the MIDDLE
# of a listed sequence, not for open-ended matching at the end.
#
# IMPORTANT: none of trip's own frames (runTestsInDir, main, stat, readdir,
# the tvm/compiler/GC code) appear as the SOURCE of these - they only show
# up as callers further down the stack. If a genuinely new bug appears
# inside trip's own code, it will NOT match these patterns and will still
# show up in results.txt as usual.
#
# After adding a new suppression, re-run `make drmemory` and check the
# "SUPPRESSIONS USED" section of results.txt - if a suppression you added
# shows 0 uses, its frame text doesn't match and needs adjusting.
#
